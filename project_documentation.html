5. SQL QUERIES

> Setup: Create a table called vendor_sales and import the CSV. All queries work in MySQL, SQL Server, and PostgreSQL (minor syntax differences noted).

Query 1: Total Sales by Vendor

SELECT 
    Vendor_Name,
    SUM(Sales_Amount) AS Total_Sales,
    SUM(Profit)       AS Total_Profit,
    COUNT(Order_ID)   AS Order_Count,
    ROUND(SUM(Profit)/SUM(Sales_Amount)*100, 2) AS Profit_Margin_Pct
FROM vendor_sales
GROUP BY Vendor_Name
ORDER BY Total_Sales DESC;
Explanation: GROUP BY Vendor_Name collapses all rows for each vendor into one summary row. SUM() adds up all their sales. ORDER BY Total_Sales DESC puts the highest seller at the top. This is the most fundamental vendor analysis query.

Query 2: Top 5 Performing Vendors

SELECT TOP 5
    Vendor_Name,
    SUM(Sales_Amount)  AS Total_Sales,
    ROUND(SUM(Profit)/SUM(Sales_Amount)*100, 2) AS Profit_Margin_Pct
FROM vendor_sales
GROUP BY Vendor_Name
ORDER BY Total_Sales DESC;
-- MySQL/PostgreSQL: Use LIMIT 5 instead of TOP 5
Explanation: TOP 5 (SQL Server) or LIMIT 5 (MySQL) restricts output to the highest 5 results. This query is useful in executive reports where only the elite performers are shown.

Query 3: Monthly Sales Trend

SELECT 
    FORMAT(Order_Date, 'yyyy-MM')  AS Month,
    SUM(Sales_Amount)              AS Monthly_Sales,
    SUM(Profit)                    AS Monthly_Profit,
    COUNT(Order_ID)                AS Orders
FROM vendor_sales
GROUP BY FORMAT(Order_Date, 'yyyy-MM')
ORDER BY Month;
-- MySQL: DATE_FORMAT(Order_Date, '%Y-%m')
-- PostgreSQL: TO_CHAR(Order_Date, 'YYYY-MM')
Explanation: FORMAT() extracts year and month from the date, grouping all orders in the same month together. This reveals seasonal patterns — for example, sales may spike in October–December due to festive season demand.

Query 4: Region-wise Performance

SELECT 
    Region,
    COUNT(Order_ID)        AS Total_Orders,
    SUM(Sales_Amount)      AS Total_Sales,
    ROUND(AVG(Discount)*100, 2) AS Avg_Discount_Pct,
    SUM(Profit)            AS Total_Profit,
    ROUND(SUM(Profit)/SUM(Sales_Amount)*100, 2) AS Profit_Margin_Pct
FROM vendor_sales
GROUP BY Region
ORDER BY Total_Sales DESC;
Explanation: Groups data by geographic region. The AVG(Discount) tells whether a region is getting higher discounts (which reduces profit). This helps the sales team prioritize regions and negotiate vendor terms geographically.

Query 5: Category Profitability Analysis

SELECT 
    Product_Category,
    SUM(Sales_Amount)      AS Total_Sales,
    SUM(Profit)            AS Total_Profit,
    ROUND(AVG(Discount)*100, 2)     AS Avg_Disc_Pct,
    ROUND(SUM(Profit)/SUM(Sales_Amount)*100, 2) AS Margin_Pct
FROM vendor_sales
GROUP BY Product_Category
ORDER BY Margin_Pct DESC;
Explanation: Finds which product categories have the best profit margins. A category with high sales but low margin (due to large discounts) may need pricing revision.

Query 6: Year-over-Year Comparison

SELECT 
    YEAR(Order_Date)   AS Year,
    SUM(Sales_Amount)  AS Total_Sales,
    SUM(Profit)        AS Total_Profit,
    COUNT(Order_ID)    AS Orders
FROM vendor_sales
GROUP BY YEAR(Order_Date)
ORDER BY Year;
Explanation: Compares 2023 vs 2024 performance at a glance. A simple but powerful query that answers the most important business question: "Are we growing?"

Vendor-level summary

vendor_summary = (
    df.groupby("Vendor_Name")
    .agg(
        Total_Sales   = ("Sales_Amount", "sum"),
        Total_Profit  = ("Profit",       "sum"),
        Orders        = ("Order_ID",     "count"),
        Avg_Discount  = ("Discount",     "mean"),

    .reset_index()
)

Click the table in the left pane → New Measure and create each: 
Total Sales = SUM(vendor_sales[Sales_Amount])

Total Profit = SUM(vendor_sales[Profit])


Profit Margin % = 
    DIVIDE([Total Profit], [Total Sales], 0) * 100


Total Orders = COUNTROWS(vendor_sales)


Avg Discount % = 
    AVERAGE(vendor_sales[Discount]) * 100


Top Vendor = 
    TOPN(1, VALUES(vendor_sales[Vendor_Name]), 
         [Total Sales], DESC)


YoY Growth % = 
    VAR current_year = CALCULATE([Total Sales], 
                        YEAR(vendor_sales[Order_Date]) = 2024)
    VAR prev_year    = CALCULATE([Total Sales], 
                        YEAR(vendor_sales[Order_Date]) = 2023)
    RETURN DIVIDE(current_year - prev_year, prev_year, 0) * 100

Section B: SQL Questions

Q6. What is the difference between WHERE and HAVING in SQL? A: WHERE filters rows before grouping. HAVING filters groups after GROUP BY has been applied. For example: WHERE Sales_Amount > 10000 filters individual rows. HAVING SUM(Sales_Amount) > 100000 filters vendor groups based on their total sales.

Q7. How would you find vendors whose total sales are above the average? A: 

SELECT Vendor_Name, SUM(Sales_Amount) AS Total_Sales
FROM vendor_sales
GROUP BY Vendor_Name
HAVING SUM(Sales_Amount) > (SELECT AVG(Total) FROM 
    (SELECT SUM(Sales_Amount) AS Total 
     FROM vendor_sales GROUP BY Vendor_Name) sub)
ORDER BY Total_Sales DESC;

Q8. What is the difference between GROUP BY and ORDER BY? A: GROUP BY collapses multiple rows with the same value into one group (used with aggregate functions like SUM, COUNT). ORDER BY sorts the result set by a column in ascending or descending order. They are often used together: GROUP BY first aggregates, then ORDER BY sorts the aggregated results.

Section C: Python Questions

Q9. Why did you use median imputation instead of mean for missing Discount values? A: The Discount column contains values between 0% and 35%, which can have a few extreme values at the boundaries. The median is the "middle value" and is not affected by these extremes, making it a more reliable representative of the typical discount rate. If we had used the mean, one or two unusual discounts could pull the imputed value too high or too low.

Q10. What is the difference between df.fillna() and df.dropna()? A: df.fillna(value) replaces missing values with a specified value (like median, mean, or 0) — it keeps all rows. df.dropna() removes any row that has at least one missing value. For this project, we used fillna() because dropping rows would reduce our dataset size (we only had 550 rows) and potentially introduce bias if missing values were not random.

Q11. What visualizations did you create and why? A: We created 12 visualizations across 3 figure panels: - Bar charts: To compare vendors and categories (good for ranking) - Pie/Donut charts: To show proportional share of regions - Line charts: To show trends over time (monthly and yearly) - Scatter plot: To show relationship between Sales, Profit, and Discount - Heatmap: To visualize the Region × Category matrix (two categorical variables) - Grouped bar: To compare Sales vs Profit side-by-side for each category

Each chart type was chosen based on the type of comparison being made (ranking, proportion, trend, relationship, or distribution).

Section D: Excel Questions

Q12. What is a Pivot Table and how did you use it? A: A Pivot Table is an Excel tool that automatically summarizes large datasets. You drag fields to "Rows", "Columns", and "Values" areas, and Excel instantly calculates aggregations like SUM, COUNT, or AVERAGE. In our project, we used Pivot Tables to summarize total sales by vendor, profit by region, and orders by month — the same analysis that required 10+ lines of Python code is done in seconds with a Pivot Table.

Q13. What Excel formula did you use to calculate profit margin, and why not hardcode it? A: We used =C3/B3 (where C3 is Profit and B3 is Total Sales), formatted as percentage. We used formulas instead of hardcoded values because Excel formulas are dynamic — if the underlying sales data changes, the profit margin automatically recalculates. Hardcoded values would need manual updating and could become incorrect. This is a core principle of spreadsheet best practice.

Section E: Power BI Questions

Q14. What is a DAX measure and how is it different from a calculated column? A: A DAX Measure is a formula that calculates values dynamically based on the current filter context (e.g., what slicer is selected). A Calculated Column adds a new column to the data table, calculated row by row at data load time. Measures are more memory-efficient and respond to slicers, making them preferred for dashboard KPIs like Total Sales or Profit Margin %.

Q15. What are slicers in Power BI and why are they important? A: Slicers are interactive filter controls — they appear as buttons, dropdowns, or checkboxes on the dashboard. When a user clicks "North" on a Region slicer, all visuals on the page automatically filter to show only North region data. Slicers transform a static report into an interactive tool, allowing business users to explore the data themselves without needing SQL or Python skills.

Q16. What is the DIVIDE function in DAX and why is it used instead of the division operator (/)? A: DIVIDE(numerator, denominator, alternate_result) safely handles division-by-zero errors by returning the alternate_result (usually 0 or BLANK()) instead of an error. The regular / operator would throw an error if the denominator is zero, which would break the entire visual. For example: DIVIDE([Total Profit], [Total Sales], 0) returns 0 if Total Sales is zero instead of crashing.
